** Unreleased

*Breaking changes*

- [[https://github.com/d12frosted/vulpea/issues/362][vulpea#362]] The org-element AST is now opt-in for extractor plugins. An extractor that reads =(vulpea-parse-ctx-ast ctx)= must declare =:requires-ast t= in its =make-vulpea-extractor= definition; without the declaration the parse context's AST slot is always nil, so a forgotten declaration fails visibly on first test (the extractor extracts nothing) instead of subtly missing inline objects in a degraded tree. Previously the conservative default meant a single registered extractor - including one that never touched the AST - forced the 2-3x slower ='object= parse for every file and disqualified every file from async extraction, and users could do nothing about a plugin that simply forgot to declare =:requires-ast nil=. Extractors that work purely from note data need no change; an explicit =:requires-ast nil= declares the intent and silences the new =vulpea-doctor= nudge about undeclared extractors. The fix is one line per AST-reading extractor. One more subtlety for code that inspects extractor structs: the =requires-ast= slot now defaults to the truthy sentinel symbol =unset=, so read it through =vulpea-extractor-requires-ast-p= (t only on an explicit declaration) instead of accessing the slot directly.

*Features*

- [[https://github.com/d12frosted/vulpea/issues/369][vulpea#369]] Exclude mentions from/to notes link the current note/linked by the current note. Set =vulpea-mentions-exclude-linked= to restore the original behaviors. That is, to not exclude mentions by existing links.
- [[https://github.com/d12frosted/vulpea/discussions/365][vulpea#365]] =vulpea-db-query-backlink-counts= now accepts a list of link types in addition to a single type, counting links of any listed type (e.g. =("id" "denote")=). Useful for collections where non-=id:= link types carry the note graph, such as denote notes migrated to vulpea with their identifiers as note IDs.
- [[https://github.com/d12frosted/vulpea/issues/359][vulpea#359]] Extractor plugins can now declare how they interact with async extraction. =make-vulpea-extractor= gains =:requires-ast=: declaring =t= hands the extract function the full org-element AST (forcing =object= granularity and the synchronous path - the AST cannot cross the process boundary), while =nil= or no declaration states the function works purely from note data, which keeps extraction at the fast =element= granularity and keeps files eligible for the worker (the extractor then runs in the main process during apply, against a context whose AST slot is always nil - see the breaking changes entry). Previously any registered extractor silently disabled the entire async pipeline and forced =object= parsing - a config with a simple attachment scanner got neither improvement. =vulpea-doctor= nudges about extractors that declare nothing, so authors state their intent explicitly. Additionally, =:worker-safe t= plus =:worker-lib= (a feature or file defining the function) lets the extractor run inside the worker in =full= mode, writing its tables through the worker's own connection; if the worker cannot resolve the function it transparently falls back to streaming results, so the extractor runs in the main process and nothing is lost. =vulpea-db-worker-rejection-reasons= makes the routing verdict inspectable.
- [[https://github.com/d12frosted/vulpea/issues/359][vulpea#359]] Add async extraction debugging tools: =vulpea-db-worker-debug= logs worker lifecycle and protocol to =*vulpea-worker-log*=; =M-x vulpea-db-worker-diagnose= runs an end-to-end health check (spawn, journal mode, mode degradations and why, whether production files actually use the worker, round-trip timing, stderr tail). A crash-loop breaker stops a worker that keeps dying right after spawn from respawning in a tight loop - it falls back to synchronous indexing with a warning (=M-x vulpea-db-worker-reset= re-arms). Full-write mode now verifies WAL journaling actually engaged and degrades to extract-only when the filesystem refuses it, since without WAL a worker write transaction would block main-process reads.
- [[https://github.com/d12frosted/vulpea/issues/359][vulpea#359]] Force re-indexing flows through the extraction worker when async extraction is enabled: parser-epoch and settings migrations (the "re-indexing all files" pass on the first activation after an upgrade) no longer freeze the session. Forced entries carry their mark through the queue, the worker protocol, crash re-queues, and stale-result retries, and skip every unchanged-content shortcut - content is identical in a migration, but extraction output is not. Without async extraction, force scans stay synchronous as before.
- [[https://github.com/d12frosted/vulpea/issues/359][vulpea#359]] Async extraction also supports a =full= mode: the worker writes results to the database through its own connection (the database switches to WAL journaling, with a busy timeout so writes from both processes interleave safely), and the main thread only registers note IDs with org-id - the save freeze disappears at any file size (~1ms measured from 1MB through 100MB). While schemas are registered with a non-silent =vulpea-db-schema-validation-action=, or any other index filter is active, =full= transparently degrades to extract-only, since those filters run in the main process. A companion =vulpea-db-async-extraction-threshold= (nil by default) can keep small files on the synchronous path for immediate query-after-save consistency; measurements show the worker also wins bulk syncs (5000 small files: 9.2s vs 10.9s wall, with half the main-thread CPU), so the threshold is about consistency, not speed.
- [[https://github.com/d12frosted/vulpea/issues/359][vulpea#359]] Add async extraction: with the new =vulpea-db-async-extraction= (off by default), the sync queue parses changed files in a persistent background =emacs --batch= worker instead of the main thread. Reading, hashing, parsing, and extraction - the bulk of indexing time - leave the UI thread entirely; only the database write of the results remains on it. Saving a 1MB file blocks Emacs for ~50ms instead of ~360ms; a 10MB file for ~0.5s instead of ~3.5s. Only extracted data crosses the process boundary (printable plists, streamed one note per line), and results flow through the exact same write path as synchronous updates - a test locks async and sync indexing to byte-identical database content. The worker mirrors an allowlist of vulpea and org settings (link types included) and, like =vulpea-db-parse-method= ='single-temp-buffer=, does not run =org-mode-hook=. Files it cannot replicate faithfully fall back to the synchronous path: anything while extractor plugins are registered (they receive the AST, which never crosses processes), function-valued =vulpea-db-index-heading-level=, and non-=.org= files. The worker starts lazily, restarts after crashes with in-flight files re-queued, discards results for files that changed mid-parse (re-queueing them), and refreshes only the stored stamp when a touched file's content hash is unchanged.

*Performance*

- [[https://github.com/d12frosted/vulpea/issues/359][vulpea#359]] Cut single-file indexing latency roughly in half. Saving a large file with autosync enabled freezes the UI for as long as that file takes to index (the sync queue is scheduled asynchronously, but the work runs on the main thread); a new benchmark (=bench/vulpea-bench-file.el=) measures that freeze across file sizes and drove three changes. With all three, the save-path freeze drops from ~630ms to ~360ms for a 1MB file and from ~6.3s to ~3.4s for a 10MB file; full scans of many small files benefit equally since the costs are per-note and per-file. The changes:
  1. Note inserts bypass emacsql's statement compilation (which profiling showed was 85% of insert time - per-value escaping in elisp, not SQLite) and go through =sqlite-execute= with native parameter binding, encoding values identically to emacsql so databases remain byte-compatible. Inserts use =OR IGNORE=, preserving the old tolerance for constraint-violating data (emacsql silently dropped such statements wholesale) but at row granularity, so one violating row no longer discards its siblings.
  2. Extraction no longer calls =org-attach-dir= once per heading (three =org-entry-get= tree walks each). When the file contains no =DIR=/=ATTACH_DIR= property - the common case, detected with a single buffer scan - the attachment directory is derived from the note's ID directly. Files that do carry these properties keep exact =org-attach-dir= semantics, inheritance included. Content hashing also stopped copying the whole buffer to a string first; digests are unchanged.
  3. The org-element parse runs at =element= granularity, skipping inline objects (bold, timestamps, entities...) that extraction never reads - 2.6x faster than a full parse. Links are located afterwards by scanning textual elements with org's own =org-element-link-parser=, so link semantics (types, positions, descriptions, exclusion of verbatim blocks) are unchanged; an adversarial corpus test locks both granularities to plist-identical output. The new =vulpea-db-parse-granularity= defaults to ='element=; set it to ='object= for the historical full parse (also forced automatically when extractor plugins are registered, since their contract is a full AST). Known difference: radio links are only picked up in ='object= mode.
- [[https://github.com/d12frosted/vulpea/issues/359][vulpea#359]] Add =vulpea-db-index-plain-links= (default =t=, no behavior change). Profiling showed that locating /unbracketed/ links - =https://...= or a bare =type:path= in running text - requires scanning all body text against =org-link-plain-re=, a large alternation over every registered link type, and that this is by far the most expensive part of link extraction. Setting the option to =nil= indexes only bracketed links, found with a cheap literal search instead: the save-path re-index of a 10MB file drops another ~20% (3.4s to 2.7s). Since =id:= links - the ones that carry the note graph - are practically always bracketed, collections that do not query plain links lose nothing. Applies identically to both parse granularities.

*Fixes*

- [[https://github.com/d12frosted/vulpea/issues/370][vulpea#370]] =vulpea-find-backlink= now lands point on the link pointing back to the current note instead of the beginning of the selected note - you see the mention itself, which is what you asked for when browsing backlinks. When the link cannot be found in the buffer (the file changed since the last sync), point stays at the note as before. Thanks to =@hpgisler= for the report.

- [[https://github.com/d12frosted/vulpea/issues/362][vulpea#362]] Extractor plugins can now actually contribute to core note fields through the documented contract. The plugin API always said an extractor receives =note-data= and returns it updated, but the returned plist was silently discarded - the note row and normalized tables were persisted before extractors ran, so a plugin adding to =:links=, =:tags= or =:meta= changed nothing, and working around it with raw SQL into core tables desynced the materialized =notes= columns from the normalized tables. The insertion order is unchanged (the note goes in first so plugin tables can hold foreign keys against it), but after all extractors run their changes to core fields are diffed and written back: the materialized column is updated and the normalized table (=tags=, =links=, =meta=, =properties=) re-synced, in every mode - synchronous, worker streaming, and worker full-write. A link contributed this way is a first-class link, visible to backlink queries and =vulpea-note-links= alike. Zero cost when no extractors are registered; identity fields (=:id=, =:path=, =:level=, =:pos=) stay fixed; an extractor returning nil means "use =note-data= as it now stands" (in-place mutations included), so plugins that end in a =when=-guarded insert stay harmless. The [[file:docs/plugin-guide.org][plugin guide]] gained a "Contributing to Core Data" section. Thanks to =@darcamo= - wiring tag-derived links in vulpea-tag-links is what surfaced the gap.

- [[https://github.com/d12frosted/vulpea/issues/359][vulpea#359]] Harden async extraction based on an adversarial multi-agent review (28 confirmed findings). The crash-respawn races are closed: a dead worker observed before its sentinel runs is salvaged (in-flight files and force marks re-enqueued) instead of silently dropped, and stale sentinels or filter output from replaced processes can no longer clobber the live worker's state. Full-write mode gained a version guard (a worker running different vulpea code streams results instead of opening - and potentially rebuilding - the database), ghost-note cleanup for files deleted mid-commit, deterministic-error surfacing instead of infinite retry, and org-id repair through stamped replies. A hang watchdog (=vulpea-db-worker-hang-timeout=, default 300s) kills a silent worker and falls back to synchronous indexing after repeated hangs. Settings changed mid-session now reach a live worker through variable watchers. Element-granularity extraction no longer indexes phantom links inside inline verbatim/code markup or macros, normalizes timestamps in meta values, and the attach-dir fast path recognizes zero-space =#+PROPERTY:DIR= and SETUPFILE-delivered properties. =M-x vulpea-db-worker-diagnose= refuses to discard in-flight work. Note for package authors: =vulpea-extractor= gained slots; packages defining extractors must be byte-recompiled against this version.

- [[https://github.com/d12frosted/vulpea/issues/356][vulpea#356]] Fix =vulpea-schema-insert-fields= inserting fields at the top of the file instead of into the note at point. With point inside a heading it now writes the fields into that heading's subtree (after the property drawer), and only falls back to file-level metadata when point is before the first heading. The synthetic note it builds while authoring is scoped the same way - it reads the note-at-point's title, tags, and existing field values from the heading rather than the file - so the missing-field computation no longer re-prompts for (and duplicate-inserts) a field the heading already carries. Thanks to =@alexjp= for the report.
- [[https://github.com/d12frosted/vulpea/issues/356][vulpea#356]] Fix the same scope mismatch in =vulpea-schema-fix-violation=. It reads the violating note at point (heading-scoped) but wrote the correction with a file-level scope by default, so applying a one-key fix to a heading-level violation could rewrite an unrelated file-level value and leave the actual violation in place. Its =BOUND= now defaults to \\='heading (matching =vulpea-schema-insert-fields=), so the fix lands in the note it was read from; pass \\='buffer to force file-level scope.
- Fix =vulpea-meta-batch-set= and =vulpea-meta-batch-remove= writing at file scope for heading-level notes. Both operated on whole-file metadata regardless of the note's level, so setting a field across a batch of heading notes inserted it at the top of each file, and removing a field could delete a same-named /file-level/ value while leaving the heading's in place - even though =vulpea-utils-process-notes= already parks point on the heading. They now scope to the heading's subtree for heading-level notes, matching the single-note =vulpea-meta-set= / =vulpea-meta-remove=. Surfaced by an audit of heading-level behaviour; the batch helpers had only ever been tested against file-level notes.
- Fix the stored position of a link in a heading's title being off by one when the heading carries a priority cookie (=[#A]=). The extractor counted =[#X]= as four characters and dropped the trailing space, so every such title-link's =:pos= was one short - enough that =vulpea-propagate-title-change= silently skipped those links (it landed on the space before the link and the match failed). TODO keywords were already handled correctly; only the priority branch was off. =vulpea-db-parser-epoch= is bumped so existing databases re-extract and pick up the corrected positions on next access.

** v2.5.0 (2026-07-02)

*Features*

- Add =vulpea-select-dyncontext-fn=, a hook for sharing expensive context across a selection. When non-nil it is called once per selection with the list of notes being presented and returns any value (a "dynamic context"); that value is passed as a second argument to =vulpea-select-describe-fn= and =vulpea-select-annotate-fn=. The second argument is only passed to functions that accept it (checked with =func-arity=), so existing one-argument describe/annotate functions keep working unchanged. This is the supported way to render data that is costly to compute per candidate - for example a backlink count: have the context function build the whole table once with =vulpea-db-query-backlink-counts= and have =describe-fn= read each note's count from it, instead of querying per candidate or wrapping the finder command to bind the hooks.
- Add =vulpea-db-query-backlink-counts=: return a hash table mapping each link destination to its number of incoming links, computed with a single grouped query. For =id:= links the destination is the target note's ID, so =(gethash (vulpea-note-id note) counts 0)= yields that note's backlink count. This is the efficient way to display or sort by backlink counts across many notes, where calling =vulpea-db-query-links-to= once per note would issue one query each. An optional =LINK-TYPE= argument (e.g. ="id"=) restricts the count to a single link type; by default every type is counted. Destinations with no incoming links are absent from the table, so read it with a default of 0.
- Expose the note's timestamps on the =vulpea-note= struct: =vulpea-note-created-at= returns the creation date derived from the =CREATED= property (normalized to a =YYYY-MM-DD= string, or nil when the note has no =CREATED= property), and =vulpea-note-modified-at= returns the file's own modification time (mtime) as a =YYYY-MM-DD HH:MM:SS= string, captured at the last sync. Neither was surfaced on query results before; they are now populated on every =vulpea-note= returned from the database (=created-at= also drives the =vulpea-db-query-by-created-*= queries). Because =modified-at= is the file's mtime, every note in a file shares it (a heading-level note carries its file's mtime), and a file edited but not yet re-synced keeps its previous value until the next sync - the same mtime that =vulpea-db-query-stale-notes= reads from the =files= table. This makes it usable for sorting or displaying notes by recency without a per-note =file-attributes= call.
- Add =vulpea-buffer-meta-change-functions=, an abnormal hook run whenever buffer metadata changes through =vulpea-buffer-meta-set=, =vulpea-buffer-meta-set-batch=, =vulpea-buffer-meta-remove= or =vulpea-buffer-meta-clean=. Each handler is called with =PROP=, =OLD= and =NEW=, where =OLD= and =NEW= are the property's lists of string values before and after the change (adding a property yields an empty =OLD=; removing one yields an empty =NEW=). It fires once per affected property and only for real changes, and it reports the outermost operation only - the nested removal that =vulpea-buffer-meta-set= performs and the clean/set churn of =vulpea-buffer-meta-sort= are coalesced, and a pure reorder fires nothing. =OLD= and =NEW= are computed only while the hook is non-nil, so an empty hook adds no overhead, and handlers run with re-entrant notification inhibited. This is the supported way to observe metadata mutations without advising the buffer API.
- [[https://github.com/d12frosted/vulpea/issues/329][vulpea#329]] Add =:target-tags= to schema =note= fields - a declarative restriction on what a link field may point at. A =note= field may now carry =:target-tags=, a list of tags the linked target note must all have; validation resolves the target and reports a new =invalid-target= violation (naming the missing tag(s)) when any are absent. The check runs per value, so it also constrains every link in a =:multiple= note field, and it only fires once the reference resolves - a missing target still reports =invalid-reference=. =link= fields are unaffected, since they are not resolved. This is the declarative form of a target restriction you previously had to hand-write in a =:validate= function; restricting a target by another schema (=:target-schema=) is not part of this change.
- [[https://github.com/d12frosted/vulpea/issues/327][vulpea#327]] Schemas can now inherit fields from one another. =vulpea-schema-define= accepts an =:include= argument - a schema name, a =vulpea-schema= object, or a list of either - whose fields are merged in before the schema's own =:fields=. Multiple includes merge left to right and the schema's own fields come last; when a =:key= appears more than once the later definition wins (so a schema overrides an inherited field, and a later include overrides an earlier one), while an overridden field keeps the position of its first appearance. Includes are resolved at definition time, so an included schema must already be registered and re-defining it later does not retroactively update its dependents; inheritance is transitive, so including a schema that itself includes others pulls in the whole chain. The merged field list is what validation and =vulpea-schema-field-value= operate on, so inherited fields behave exactly like native ones. The predicate is not inherited - each schema declares its own. As a side effect of always running the merge, a duplicate =:key= within a single schema's own =:fields= now collapses to its last definition (previously both entries were kept and validated twice).
- [[https://github.com/d12frosted/vulpea/issues/330][vulpea#330]] Add schema-driven note authoring. =vulpea-schema-insert-fields= is an interactive command that inserts the fields of an applicable schema into the current buffer: it resolves the schema from the buffer (prompting when several apply, or over all registered schemas when none do), then prompts for each field the note does not already have, required ones first, offering =:one-of= values as completion and a note selection for =note= fields, and inserts the answers as metadata. A =note= value is stored as a proper =id:= link, existing fields are never overwritten, and with a prefix argument it skips prompting and inserts empty placeholders instead. Because it operates on a schema's merged fields, inherited (=:include=) fields are offered too. Two headless primitives back the command and are useful on their own: =vulpea-schema-applicable= returns the names of schemas whose predicate matches a note, and =vulpea-schema-missing-fields= returns the fields a note lacks (required ones first).
- [[https://github.com/d12frosted/vulpea/issues/331][vulpea#331]] Surface schema violations as notes are indexed. =vulpea-db-schema-validation-action= controls what happens when a note that an applicable schema matches fails validation while it is written to the database (on save, external sync, or a full scan): =silent= indexes it quietly, =warning= (the default) indexes it and emits a non-blocking warning naming the violations, and =error= keeps the violating note out of the database entirely and warns. Because files are the source of truth this never touches the file - =error= simply leaves the note absent from the database until it is fixed and re-synced. The check is a no-op unless schemas are registered and one applies to the note. It is built on a new general extension point - the =vulpea-db-note-index-filter-functions= abnormal hook, run per note in =vulpea-db-update-file= so a note is indexed only when every handler returns non-nil - and a new engine primitive =vulpea-schema-note-violations= that validates a note against every applicable schema.
- [[https://github.com/d12frosted/vulpea/issues/331][vulpea#331]] Add in-buffer schema linting. =vulpea-schema-flymake-mode= is a buffer-local minor mode (off by default) that flags schema violations in a note buffer with flymake as you edit: every note in the buffer (file-level and heading-level) is validated against each applicable schema, and each violation is shown on the offending field's metadata line, or on the note's heading for a missing required field. Enable it per buffer or from a hook (for example on =org-mode-hook=). It reuses =vulpea-schema-note-violations=, so it is the live, in-buffer counterpart to the sync-time =vulpea-db-schema-validation-action=.
- [[https://github.com/d12frosted/vulpea/issues/342][vulpea#342]] Add =vulpea-schema-fix-violation=, a headless one-shot fix for a =vulpea-violation=. It resolves the violated field from the violation's schema, prompts for a corrected value the same way =vulpea-schema-insert-fields= does (=:one-of= completion, note selection for =note= fields, =:multiple=), and writes it - replacing the offending value, or inserting the field when it was missing - returning the value written or nil when skipped. This is the building block a UI uses to offer one-key fixes from a validation view (e.g. the schema health widget in vulpea-ui). As part of it, note-field prompting now honours =:target-tags=: both =vulpea-schema-insert-fields= and the fixer restrict the note selection to notes that carry every required tag, so you can only pick a valid target.
- Add =vulpea-schema-collection-health=, the collection-wide counterpart to =vulpea-schema-note-violations=. In a single pass over a list of notes (defaulting to all notes in the database) it returns one =vulpea-schema-health= structure per registered schema, in registration order, reporting how many notes the schema covers (its predicate matches), how many of those are invalid, the invalid notes paired with their violations (=(note . violations)=), and the schema's include relationships in both directions (=includes= and =included-by=). Schemas that match no notes are still reported with zero counts, so the result is a faithful map of every registered schema - the data source a schema dashboard renders. As part of this a =vulpea-schema= now retains the names of the schemas it includes, exposed as =vulpea-schema-includes= (previously =:include= was folded into the merged field list and otherwise discarded).
- [[https://github.com/d12frosted/vulpea/pull/325][vulpea#325]] Add =vulpea-db-exclude-property=, the drawer property that marks a note (file-level or heading-level) for exclusion from the database. It defaults to =VULPEA_IGNORE=, so nothing changes out of the box, but you can point it at any property name - for example =ROAM_EXCLUDE= to reuse the exclusion marks of an existing org-roam collection. The value is now read through =org-not-nil=, matching how Org reads its own boolean drawer properties (=ORDERED=, =UNNUMBERED=, =COPYING=): any value other than =nil= excludes the node, so it can even carry a human-readable reason (=:VULPEA_IGNORE: private, keep this out=), while =:VULPEA_IGNORE: nil= no longer excludes. Previously the mere presence of the property excluded the node regardless of value, so an explicit =nil= still hid it. Thanks to =@noncog= for the feature and the fix.

*Fixes*

- Ignore =#+KEYWORD:= lines that live inside verbatim blocks when reading buffer properties. =vulpea-buffer-prop-get= and =vulpea-buffer-prop-get-all= (and =vulpea-buffer-tags-get=, which builds on them) scanned line-wise and would pick up a =#+filetags:= - or any =#+KEY:= - quoted inside a =#+begin_src= / =#+begin_example= block as if it were a real document keyword. A note that merely quotes Org markup (for example a post about filetags) would then wrongly inherit those tags. The readers now consult =org-element= and skip matches inside verbatim blocks (source, example, export, comment), so only genuine top-level keywords are returned. The database extractor was already immune - it walks the =org-element= AST rather than scanning text - and a regression test now locks that in.
- Extend that verbatim-block awareness to the buffer property /writers/. =vulpea-buffer-prop-set=, =vulpea-buffer-prop-remove= and =vulpea-buffer-prop-remove-all= also scanned line-wise, so a =#+KEY:= quoted inside a =#+begin_src= / =#+begin_example= block could be rewritten or deleted as if it were a real keyword. This surfaced through =vulpea-buffer-tags-set=, which consolidates filetags by removing every =#+filetags:= line and re-adding one: on a note that quotes =#+filetags:= inside a block it wiped the quoted example, and could even relocate the real tags into the block. All three writers now skip matches inside verbatim blocks (source, example, export, comment). When a keyword exists only inside a block, =vulpea-buffer-prop-set= inserts a fresh line instead of overwriting the quoted one.
- Populate the =closed= field from the =:LOGBOOK:= drawer when a task has no =CLOSED:= planning line. A heading-level note only carries a =CLOSED:= planning line when =org-log-done= adds one; the very common setup that instead logs state changes into a =:LOGBOOK:= drawer (=- State "DONE" from "TODO" [timestamp]=) left =closed= empty, so the database column was effectively unpopulated and queries over it (e.g. "tasks completed on a date") returned almost nothing. Extraction now falls back to the most recent =State "DONE"= log entry in the note's LOGBOOK drawer when the planning =CLOSED:= is absent. The fallback only applies to notes currently in a done state, mirroring how Org drops =CLOSED= when a task is reopened, and a =CLOSED:= planning line still takes precedence when present. It keys on =State "DONE"= specifically, so a cancelled task (logged as =State "CANCELLED"=) is left without a =closed= value. The =vulpea-db-parser-epoch= is bumped so existing databases re-extract every file once on next access and backfill =closed= automatically, without a manual scan or discarding the database.
- [[https://github.com/d12frosted/vulpea/issues/344][vulpea#344]] Fix external sync never triggering on Windows with =fswatch=. The paths =fswatch= reported did not match the watched directories vulpea compares them against, so every external change (and even in-Emacs saves) was silently ignored until the next full scan. The MSYS2/mingw build reports an upper-case drive letter and a backslash before the file name (e.g. =C:/notes\bar.org=), while =expand-file-name= yields a lower-case drive and forward slashes (=c:/notes/bar.org=), so the =string-prefix-p= check always failed. fswatch paths are now normalized before the directory check - backslashes to forward slashes, drive letter lower-cased, and a =/cygdrive/= prefix rewritten to a drive - and the check itself is case-insensitive on Windows. Thanks to =@egorelik93= for the report and for pinning down the exact path formats both builds emit.

- Always reset the shared parse buffer's state after parsing a file, even when extraction errors.   - =vulpea-meta-get!= - function to get a value of =PROP= from =META= (result of =vulpea-meta=). Value is parsed based on the passed =TYPE= or as a string if omitted. Use it performing multiple read operations in a row.
  - =vulpea-meta-get-list= - function to get all values of =PROP= for note with =ID=. Values are parsed based on the passed =TYPE= or as a string if omitted.
  - =vulpea-meta-get-list!= - function to get all values of =PROP= from =META= (result of =vulpea-meta=). Values are parsed based on the passed =TYPE= or as a string if omitted. Use it performing multiple read operations in a row.
  - =vulpea-meta-set= - function to set =VALUE= of =PROP= for =NOTE-OR-ID=. Supports multi-value properties.
  - =vulpea-meta-remove= - function to remove a =PROP= for =NOTE-OR-ID=.
  - =vulpea-meta-remove= - function to remove all meta for =NOTE-OR-ID=.
  - =vulpea-meta-format= - function to format a =VALUE=.
- =vulpea-utils= module.
  - =vulpea-note= type definition.
  - =vulpea-utils-with-note= - function to execute =BODY= with point at =NOTE=. Supports file-level notes as well as heading notes.
  - =vulpea-utils-link-make-string= - make a bracket link to =NOTE=.
