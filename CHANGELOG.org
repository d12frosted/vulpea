** Unreleased

*Features*

- Add =vulpea-select-dyncontext-fn=, a hook for sharing expensive context across a selection. When non-nil it is called once per selection with the list of notes being presented and returns any value (a "dynamic context"); that value is passed as a second argument to =vulpea-select-describe-fn= and =vulpea-select-annotate-fn=. The second argument is only passed to functions that accept it (checked with =func-arity=), so existing one-argument describe/annotate functions keep working unchanged. This is the supported way to render data that is costly to compute per candidate - for example a backlink count: have the context function build the whole table once with =vulpea-db-query-backlink-counts= and have =describe-fn= read each note's count from it, instead of querying per candidate or wrapping the finder command to bind the hooks.
- Add =vulpea-db-query-backlink-counts=: return a hash table mapping each link destination to its number of incoming links, computed with a single grouped query. For =id:= links the destination is the target note's ID, so =(gethash (vulpea-note-id note) counts 0)= yields that note's backlink count. This is the efficient way to display or sort by backlink counts across many notes, where calling =vulpea-db-query-links-to= once per note would issue one query each. An optional =LINK-TYPE= argument (e.g. ="id"=) restricts the count to a single link type; by default every type is counted. Destinations with no incoming links are absent from the table, so read it with a default of 0.
- Expose the note's timestamps on the =vulpea-note= struct: =vulpea-note-created-at= returns the creation date derived from the =CREATED= property (normalized to a =YYYY-MM-DD= string, or nil when the note has no =CREATED= property), and =vulpea-note-modified-at= returns the =YYYY-MM-DD HH:MM:SS= timestamp recorded when the note was last synced into the database. Both values were already materialized in the =notes= table (and drive the =vulpea-db-query-by-created-*= queries) but were not surfaced on query results; they are now populated on every =vulpea-note= returned from the database. Note that =modified-at= tracks indexing time, not the file's own modification time - for the latter use =vulpea-db-query-stale-notes=, which reads the =files= table.
- Add =vulpea-buffer-meta-change-functions=, an abnormal hook run whenever buffer metadata changes through =vulpea-buffer-meta-set=, =vulpea-buffer-meta-set-batch=, =vulpea-buffer-meta-remove= or =vulpea-buffer-meta-clean=. Each handler is called with =PROP=, =OLD= and =NEW=, where =OLD= and =NEW= are the property's lists of string values before and after the change (adding a property yields an empty =OLD=; removing one yields an empty =NEW=). It fires once per affected property and only for real changes, and it reports the outermost operation only - the nested removal that =vulpea-buffer-meta-set= performs and the clean/set churn of =vulpea-buffer-meta-sort= are coalesced, and a pure reorder fires nothing. =OLD= and =NEW= are computed only while the hook is non-nil, so an empty hook adds no overhead, and handlers run with re-entrant notification inhibited. This is the supported way to observe metadata mutations without advising the buffer API.
- [[https://github.com/d12frosted/vulpea/issues/329][vulpea#329]] Add =:target-tags= to schema =note= fields - a declarative restriction on what a link field may point at. A =note= field may now carry =:target-tags=, a list of tags the linked target note must all have; validation resolves the target and reports a new =invalid-target= violation (naming the missing tag(s)) when any are absent. The check runs per value, so it also constrains every link in a =:multiple= note field, and it only fires once the reference resolves - a missing target still reports =invalid-reference=. =link= fields are unaffected, since they are not resolved. This is the declarative form of a target restriction you previously had to hand-write in a =:validate= function; restricting a target by another schema (=:target-schema=) is not part of this change.
- [[https://github.com/d12frosted/vulpea/issues/327][vulpea#327]] Schemas can now inherit fields from one another. =vulpea-schema-define= accepts an =:include= argument - a schema name, a =vulpea-schema= object, or a list of either - whose fields are merged in before the schema's own =:fields=. Multiple includes merge left to right and the schema's own fields come last; when a =:key= appears more than once the later definition wins (so a schema overrides an inherited field, and a later include overrides an earlier one), while an overridden field keeps the position of its first appearance. Includes are resolved at definition time, so an included schema must already be registered and re-defining it later does not retroactively update its dependents; inheritance is transitive, so including a schema that itself includes others pulls in the whole chain. The merged field list is what validation and =vulpea-schema-field-value= operate on, so inherited fields behave exactly like native ones. The predicate is not inherited - each schema declares its own. As a side effect of always running the merge, a duplicate =:key= within a single schema's own =:fields= now collapses to its last definition (previously both entries were kept and validated twice).
- [[https://github.com/d12frosted/vulpea/issues/330][vulpea#330]] Add schema-driven note authoring. =vulpea-schema-insert-fields= is an interactive command that inserts the fields of an applicable schema into the current buffer: it resolves the schema from the buffer (prompting when several apply, or over all registered schemas when none do), then prompts for each field the note does not already have, required ones first, offering =:one-of= values as completion and a note selection for =note= fields, and inserts the answers as metadata. A =note= value is stored as a proper =id:= link, existing fields are never overwritten, and with a prefix argument it skips prompting and inserts empty placeholders instead. Because it operates on a schema's merged fields, inherited (=:include=) fields are offered too. Two headless primitives back the command and are useful on their own: =vulpea-schema-applicable= returns the names of schemas whose predicate matches a note, and =vulpea-schema-missing-fields= returns the fields a note lacks (required ones first).

** v2.4.0

*Features*

- [[https://github.com/d12frosted/vulpea/issues/210][vulpea#210]] Add =vulpea-note-unlinked-mentions-async=: find notes that mention a note's title or aliases as plain text without an =id:= link to it. The scan uses ripgrep over =vulpea-db-sync-directories=, excludes occurrences already inside an Org link, hits in the note's own file or in the file of a note that shares its title (a title collision), and matches on Org metadata lines (keywords like =#+title:= and property-drawer lines, which are a note's own metadata rather than prose), and maps each remaining hit to the mentioning note. It is asynchronous and promise-style - it calls one of two =resolve= / =reject= callbacks - so it plugs directly into reactive UIs (e.g. as a =vui-use-async= loader). It returns the ripgrep process (or nil when it answers synchronously) so a caller can wait on or cancel it. Requires =rg= on =exec-path=.
- [[https://github.com/d12frosted/vulpea/issues/210][vulpea#210]] Add =vulpea-buffer-unlinked-mentions-async= - the outgoing counterpart: find existing notes that the current buffer mentions as plain text without an =id:= link to them. It searches the buffer's live content (so unsaved edits are included) with ripgrep for the titles and aliases of the candidate notes, excludes occurrences inside an Org link, on a metadata line, or for a note in the buffer's own file, and returns the candidate note(s) each remaining match could link to. The candidate notes are chosen by =vulpea-mentions-note-filter= (default: file-level notes only, which avoids noise from heading-level notes). Like the incoming function it is async/promise-style and returns the ripgrep process.
- =vulpea-doctor= now lists =ripgrep= (=rg=) availability and its path in the External Tools section, alongside =fd=, =fswatch=, and =git=.
- [[https://github.com/d12frosted/vulpea/issues/146][vulpea#146]] Add =vulpea-tags-expand= for hierarchical tag queries: given a list of tags, it returns that list plus the transitive members of any Org tag groups among them (from =org-tag-groups-alist=). Combine it with the tag query functions to match a parent tag and all its descendants, e.g. =(vulpea-db-query-by-tags-some (vulpea-tags-expand '("GTD")))=. Meaningful for the =-some= and =-none= queries; not applied to =-every= (where "has all of" a group is a different question). Cyclic group definitions terminate safely.
- [[https://github.com/d12frosted/vulpea/issues/214][vulpea#214]] Add a schema validation engine: =vulpea-schema-validate= checks a note against a schema and returns a list of =vulpea-violation= structs; =vulpea-schema-validate-notes= validates an arbitrary list of notes; =vulpea-schema-validate-all= validates every note a schema's predicate matches. Violation types are =missing-required=, =wrong-type=, =invalid-reference= (a =note= field pointing at a non-existent note), =disallowed-value= (not in =:one-of=), and =invalid-value= (rejected by a =:validate= function). The per-note and list forms validate exactly what they are given; only =validate-all= consults the predicate.
- [[https://github.com/d12frosted/vulpea/issues/213][vulpea#213]] Add a schema definition system (=vulpea-schema=): declare structured expectations for a class of notes - a predicate selecting which notes a schema applies to, plus field specs declaring metadata keys with types, required-ness, and allowed values. =:required= and =:one-of= accept a function of the note for conditional rules and dependent enums. Includes the registry (=vulpea-schema-define=, =vulpea-schema-get=, =vulpea-schema-list=, =vulpea-schema-unregister=), =vulpea-schema-applies-p=, and =vulpea-schema-field-value=. Validation against a schema is added separately.
- [[https://github.com/d12frosted/vulpea/issues/215][vulpea#215]] Add =vulpea-db-query-stale-notes=: return notes whose file has not been modified within the last N days, using the modification time recorded in the =files= table during sync. All notes in a stale file are returned (heading-level notes share their file's mtime), ordered oldest file first - useful for surfacing notes that may need review or archiving.
- [[https://github.com/d12frosted/vulpea/issues/277][vulpea#277]] =vulpea-doctor= now reports =cached files= (how many files the change-detection cache tracks) and =files without notes= (how many of those produced no note) in its Database section. A surprising =files without notes= count is a quick signal for notes stuck in the cache and missing from the database; re-extract them with =C-u M-x vulpea-db-sync-full-scan=.
- [[https://github.com/d12frosted/vulpea/issues/166][vulpea#166]] Add a =:create-fn= argument to =vulpea-find=, and a =vulpea-find-default-create-fn= global default, controlling how a note is created when the user selects a non-existent note. It is called with the typed title and capture properties (currently always nil), mirroring the existing =:create-fn= of =vulpea-insert=, and should return the =vulpea-note= to visit (or nil to skip visiting). This is the hook for "capture on empty" workflows - point it at =org-capture= or a custom command to turn a search that found nothing straight into note creation. The previous behavior (create from the title via =vulpea-create=) is unchanged and is now the default value of =vulpea-find-default-create-fn=.
- [[https://github.com/d12frosted/vulpea/issues/166][vulpea#166]] Add =vulpea-insert-default-create-fn=, a global "capture on empty" default for =vulpea-insert= mirroring =vulpea-find-default-create-fn=. When it is set and no explicit =:create-fn= is passed, it is called with the typed title and capture properties for a non-existent note. Unlike the =vulpea-find= hook it must insert the link itself, since inserting a link is what =vulpea-insert= does with a new note. Defaults to nil, i.e. the built-in behavior (create via =vulpea-create= and insert an id: link).
- [[https://github.com/d12frosted/vulpea/issues/166][vulpea#166]] Note selection now reports a =vulpea-note= completion category. =vulpea-select-from= (and therefore =vulpea-select=, =vulpea-find=, and =vulpea-insert=) wraps its candidates in a completion table whose metadata sets =(category . vulpea-note)=, so completion UIs and their extensions can recognize note candidates: marginalia can annotate them, embark can offer note actions (e.g. embark-vulpea, which keys on this category and the =vulpea-note-id= text property already attached to each candidate), and consult integrations can group or preview them. The change is additive - plain =completing-read= is unaffected.

*Fixes*

- =vulpea-create= now inserts heading-level notes with deterministic spacing. A heading appended after a sibling subtree is separated from it by exactly one blank line, the first child of a parent (or the first heading in a file) gets none, and the file keeps a single trailing newline. Trailing whitespace from a =:body= no longer leaks into the file. Previously, appending heading-level notes - especially ones carrying a =:body= - could accumulate several blank lines before the next top-level heading and at the end of the file.
- Document that =vulpea-create-default-function= and =vulpea-create-default-template= seed file-level note creation only. When =vulpea-create= is called with a =:parent= (a heading-level note), no defaults are consulted and the heading is built solely from the explicitly passed arguments. Clarified in the docstrings and in the configuration and user guides; behavior is unchanged.
- [[https://github.com/d12frosted/vulpea/issues/277][vulpea#277]] Invalidate the file change-detection cache when the note extractor changes, via a new parser epoch (=vulpea-db-parser-epoch=). Previously a file that was once indexed while producing no note (e.g. a partially-synced cloud file, or an older Vulpea version that failed to extract its =ID=) had that "no note" result cached, so later scans kept skipping it and the note stayed invisible until the file was edited or a force scan was run. Bumping the parser epoch now clears the cache and re-extracts every file once, without discarding the database. Existing databases predating this mechanism are treated as stale and re-extracted on first open, so the fix applies retroactively on upgrade.
- [[https://github.com/d12frosted/vulpea/issues/277][vulpea#277]] Upcase =vulpea-buffer-alias-property= before looking it up during extraction. Property keys are stored upcased, so a lowercase or mixed-case alias property name (e.g. =(setq vulpea-buffer-alias-property "roam_aliases")=) previously caused aliases to be silently dropped. The default =ALIASES= and the common =ROAM_ALIASES= were unaffected.
- [[https://github.com/d12frosted/vulpea/issues/301][vulpea#301]] Stop warning =Making delay-mode-hooks buffer-local while locally let-bound!= when the shared parse buffer is initialized. The buffer setup let-bound =delay-mode-hooks= around =org-mode=, but =org-mode= itself makes that variable buffer-local, which is not allowed while it is let-bound. Use the =delay-mode-hooks= macro instead, which is the idiomatic way to skip mode hooks. Behavior is unchanged.

** v2.3.0

*Features*

- [[https://github.com/d12frosted/vulpea/issues/277][vulpea#277]] Change the default of =vulpea-db-sync-scan-on-enable= from =nil= to =async=: enabling autosync now scans note directories in the background, so changes made while Emacs was closed (git pulls, cloud sync, external edits) are picked up automatically instead of staying invisible until the files are touched again. Additionally, when the database is empty (e.g. the very first activation), an async scan is performed even when the option is set to =nil= - otherwise the database would stay empty with no indication why. Users of very large collections (10000+ notes) who prefer the old behavior can set the option back to =nil=.
- [[https://github.com/d12frosted/vulpea/issues/277][vulpea#277]] Add =vulpea-doctor= - an interactive diagnostics command that reports versions, configuration, database state, sync state, and external tool availability in a single buffer, and flags common setup problems: missing or empty sync directories, an empty or missing database, =fswatch= / =fd= not visible to Emacs (a frequent =PATH= issue on GUI Emacs and Doom), autosync disabled, and external monitoring configured but not running. The command is read-only and modifies nothing. Please include its output in bug reports.
- [[https://github.com/d12frosted/vulpea/issues/277][vulpea#277]] Add =vulpea-version= - a constant with the release version and an interactive function reporting the most precise version available: =git describe= output when running from a git checkout (e.g. =v2.2.0-15-g2938416=), the installed package version including the commit for MELPA snapshot installs (e.g. =20260610.1234 (commit 2938416)=), with the constant as a fallback. Please include its output in bug reports.
- [[https://github.com/d12frosted/vulpea/issues/270][vulpea#270]] Add =vulpea-db-sync-verbose= (default =t=) to control routine sync status reporting in the echo area. When set to =nil=, messages such as =Vulpea: Syncing N files...= and =Vulpea: Sync complete= are suppressed — useful to avoid distraction on every save when autosync is enabled. Errors and warnings are always shown; low-level timing diagnostics remain controlled by =vulpea-db-sync-debug=.
- [[https://github.com/d12frosted/vulpea/issues/251][vulpea#251]] Automatically detect when global tag inheritance settings (=org-use-tag-inheritance=, =org-tags-exclude-from-inheritance=) change between sessions and trigger a forced re-index. A fingerprint of these settings is stored in the database; on startup, if the fingerprint differs, the file change cache is cleared so all files are re-extracted with the new settings. Per-buffer overrides (e.g. via =.dir-locals.el=) are not tracked — use =vulpea-db-sync-update-directory= with a force argument after changing those.

*Documentation*

- Add a ready-made Doom Emacs module in the =doom/= directory (install as =:tools vulpea= by symlinking it to =$DOOMDIR/modules/tools/vulpea=). It installs vulpea via =package!=, enables =vulpea-db-autosync-mode= on first input after startup, binds the main commands under =SPC n v=, and adds =doom doctor= checks that warn when =fswatch= / =fd= are not visible to Emacs (the stale =doom env= pitfall). See =doom/README.org=.

- Expand the README ecosystem section with third-party packages (=embark-vulpea=, =vulpea-field=, =publicatorg=) and grow the real-world usage section with applications built on Vulpea (=claude-orgmode=, =emacs-org-serve=) and notable user configurations.

*Fixes*

- Always reset the shared parse buffer's state after parsing a file, even when extraction errors. Previously a parse error left =buffer-file-name= pointing at the file on the reused parse buffer, which could mark it modified and interfere with file-change tracking on a later save. The cleanup now runs in an =unwind-protect=.
- Make =vulpea-buffer-prop-get= bind =case-fold-search= (like its sibling readers) so a property is found regardless of the caller's setting, and quote the property name so a regexp-special character in it is matched literally rather than as part of the search pattern.
- Match directory paths literally in =vulpea-db-query-by-directory=. It built a SQL =LIKE= pattern from the directory name without escaping, so =%= or =_= in a path (e.g. =my_notes=) acted as wildcards and pulled in sibling directories. It now uses =GLOB= with the directory escaped for GLOB's special characters, sharing the escape helper with the sync layer (=vulpea-db--escape-glob-pattern=). Note that matching is now case-sensitive, consistent with file-path semantics.
- Make =vulpea-db-query-by-tags-every= and =vulpea-db-query-by-links-every= ignore duplicate entries in their input. Because they match on =COUNT(DISTINCT ...)= equal to the number of requested items, a repeated tag or id previously inflated the target count and made the query return nothing; the input is now de-duplicated first.
- Insert titles and property values containing backslashes verbatim. =vulpea-buffer-prop-set= (and thus =vulpea-buffer-title-set=) and the link-description update used by =vulpea-propagate-title-change= called =replace-match= without the LITERAL flag, so a value containing =\1=, =\&= or =\\= was interpreted as a regexp backreference and corrupted the result. They now replace literally.
- Stop =vulpea-note-meta-get= / =vulpea-note-meta-get-list= with type =note= from raising =wrong-type-argument= when a metadata id-link points to a note that no longer exists. A dangling link now yields a nil entry instead of crashing, matching the behavior of the buffer-level meta API.
- Restore compatibility with Emacs 27-28 for batch operations. =vulpea-utils-process-notes= (used by =vulpea-tags-batch-*= and =vulpea-meta-batch-*=) called =kill-matching-buffers-no-ask=, which only exists since Emacs 29.1, so every batch command failed with a void-function error on the supported Emacs 27.2-28. It now uses that function when available and falls back to an internal no-prompt buffer sweep on older Emacsen.
- Build the internal =sqlite_master= existence checks (=vulpea-db--table-exists-p=, =vulpea-db--index-exists-p=) with emacsql parameters instead of string interpolation, so an object name can never break out of the query. These are only called with internal identifiers today, so this is hardening rather than a user-facing fix.
- Security: prevent arbitrary Elisp execution during note creation. In =vulpea-create= templates, the =%(elisp)= and =%<format>= directives are now expanded on the template itself /before/ =${title}=, =${slug}=, =${id}= and =:context= values are substituted in. Previously the substitution happened first, so a value that happened to contain =%(...)= - most notably a note title (which can come from typed input, the active region, or other untrusted sources via =vulpea-find= / =vulpea-insert=) - was evaluated as code. Directives written by the template author keep working exactly as before; substituted values are now always treated as literal data.
- [[https://github.com/d12frosted/vulpea/issues/275][vulpea#275]] Make external file monitoring setup idempotent. =vulpea-db-sync--setup-fswatch= and =vulpea-db-sync--setup-polling= now no-op when a process or timer is already running, so repeated activation no longer spawns orphaned =fswatch= processes or leaks polling timers. Orphaned =fswatch= processes were particularly persistent because the sentinel re-launches them on exit; legitimate sentinel-driven restarts are unaffected, since the sentinel clears the process handle before restarting.
- [[https://github.com/d12frosted/vulpea/issues/271][vulpea#271]] Create the parent directory of =vulpea-db-location= during database initialization if it does not exist. Previously, pointing =vulpea-db-location= at a not-yet-existing subdirectory caused emacsql to fail with an opaque =(sqlitep nil)= error.
- [[https://github.com/d12frosted/vulpea/pull/267][vulpea#267]] Fix =Wrong type argument: hash-table-p= error during async sync. When =org-id-update-id-locations= is running, =org-id-locations= is temporarily an alist. If a vulpea timer fires in that window, =org-id-add-location= fails. Now =org-id-locations= is defensively converted to a hash table before registering IDs.
- Fix =org-element= parser errors (="Invalid search bound"=, =wrong-type-argument integer-or-marker-p=) when building the database from scratch. The reusable parse buffer replaces its contents with =inhibit-modification-hooks= set, leaving the =org-element= cache stale from the previous file. Now =org-element-cache-reset= is called after each buffer replacement.

** v2.2.0

*Features*

- [[https://github.com/d12frosted/vulpea/issues/243][vulpea#243]] Automatic database rebuild on schema version change. When =vulpea-db-version= changes, the database file is deleted and recreated from scratch on next access. When =vulpea-db-autosync-mode= is active, a forced re-index of all files is triggered automatically.
- [[https://github.com/d12frosted/vulpea-journal/issues/3][vulpea-journal#3]] Add =vulpea-db-extra-extensions= for tracking non-.org files (e.g., =.org.age=, =.org.gpg=). Files with extra extensions are always parsed via =find-file= to support decryption hooks. Default is =nil= (opt-in only).
- Store link descriptions in the database. Links now include a =:description= field extracted during parsing, eliminating the need for file I/O when querying link descriptions. The =vulpea-db-query-links-*= functions now return plists with =:source=, =:dest=, =:type=, =:pos=, and =:description= keys. Database schema version bumped to 3 (existing databases will rebuild automatically).
- [[https://github.com/d12frosted/vulpea/issues/212][vulpea#212]] Add title propagation system for renaming notes:
  - =vulpea-propagate-title-change= - interactive command to rename a note's title and update incoming link descriptions. Supports dry-run with =C-u= prefix, categorizes links into exact matches (auto-update) and partial matches (manual review), and optionally renames the file.
  - =vulpea-rename-file= - function to rename a note's file based on a new title slug and update the database.

** v0.1
:PROPERTIES:

*Features*

- =vulpea= module - one stop module, importing all others.
  - =vulpea-select= - function to =completing-read= a note with optional filter.
  - =vulpea-create= - function to create a new note file with given =TITLE= and =TEMPLATE=.
- =vulpea-db= module - for querying notes data base.
  - =vulpea-db-get-id-by-file= - function to get =ID= of a note represented by =FILE=.
  - =vulpea-db-get-by-id= - function to get note represented by =ID=. Supports headings of the note.
  - =vulpea-db-get-file-by-id= - function to get =FILE= of a note represented by =ID=. Supports headings of the note.
  - =vulpea-db-search-by-title= - function to query notes with =TITLE=.
  - =vulpea-db-update= - function to update db with =NOTE-OR-ID=.
- =vulpea-meta= module - for manipulating note metadata represented by description list:
  - =vulpea-meta= - function to get metadata from =NOTE-OR-ID=. In most cases you should not use this function unless performance is important. In this case, take a look at bang functions, e.g. =vulpea-meta-get!=.
  - =vulpea-meta-get= - function to get a value of =PROP= for note with =ID=. Value is parsed based on the passed =TYPE= or as a string if omitted.
  - =vulpea-meta-get!= - function to get a value of =PROP= from =META= (result of =vulpea-meta=). Value is parsed based on the passed =TYPE= or as a string if omitted. Use it performing multiple read operations in a row.
  - =vulpea-meta-get-list= - function to get all values of =PROP= for note with =ID=. Values are parsed based on the passed =TYPE= or as a string if omitted.
  - =vulpea-meta-get-list!= - function to get all values of =PROP= from =META= (result of =vulpea-meta=). Values are parsed based on the passed =TYPE= or as a string if omitted. Use it performing multiple read operations in a row.
  - =vulpea-meta-set= - function to set =VALUE= of =PROP= for =NOTE-OR-ID=. Supports multi-value properties.
  - =vulpea-meta-remove= - function to remove a =PROP= for =NOTE-OR-ID=.
  - =vulpea-meta-remove= - function to remove all meta for =NOTE-OR-ID=.
  - =vulpea-meta-format= - function to format a =VALUE=.
- =vulpea-utils= module.
  - =vulpea-note= type definition.
  - =vulpea-utils-with-note= - function to execute =BODY= with point at =NOTE=. Supports file-level notes as well as heading notes.
  - =vulpea-utils-link-make-string= - make a bracket link to =NOTE=.
